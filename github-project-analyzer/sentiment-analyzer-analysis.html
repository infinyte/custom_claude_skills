Project Purpose & Scope

Sentiment Analyzer is a real-time cryptocurrency sentiment analysis and multi-agent reinforcement learning (MARL) trading platform. It aggregates market data from CoinGecko, ingests sentiment signals from news (NewsAPI) and 7 social media sources (Twitter/X, Reddit, RSS, Discord, Telegram, YouTube, TikTok), and uses Claude AI plus local NLP heuristics to generate Bull/Neutral/Bear sentiment assessments for top altcoins. A sophisticated evolutionary trading system allows multiple AI agents to compete in sandboxed tournaments, discovering emergent trading strategies through adversarial Q-learning and genetic algorithms.

Current State & Maturity

The project is in active beta development. All three major phases are complete:

  • Phase 1 (Sentiment & Dashboard): Core sentiment analysis, React dashboard with 7 tabs, polling-based updates.
  • Phase 2 (MARL Evolutionary Trading): Multi-agent competition engine, genetic algorithms, agent lifecycle management, persisted win-rate leaderboards, and real broker integration (Crypto.com, Binance.US, Coinbase, Alpaca).
  • Phase 3 (Social Intelligence): 7-source scraper suite, 4-signal scoring pipeline, trending discovery engine, and per-symbol trend reports with velocity tracking.

Development is continuous. The codebase undergoes regular updates with CI/CD via GitHub Actions, type-safe build process, and a well-maintained test suite.

Development Pace & Signals

Commit activity: Multiple commits per week typical for active feature work and bug fixes. Recent work includes live tournament monitoring via SSE, Phase 2 MARL gap closure, Q-value persistence, evolutionary tracking, and social media enhancements.

Community & Usage: Private repository under the "infinyte" GitHub organization. The project serves as both a personal research platform and a working trading intelligence tool, suggesting serious long-term commitment.

Intended Audience & Use Case

Primary audience: Cryptocurrency traders, algorithmic trading researchers, and AI/ML practitioners interested in multi-agent systems and reinforcement learning applied to financial markets.

Key use cases:

  • Real-time sentiment monitoring across 50+ altcoins with multi-source intelligence
  • Paper and live trading via multiple exchanges with zero-trust architecture
  • Agent breeding, evolution, and genealogy tracking for emergent strategy discovery
  • Backtesting and pre-training of agents on synthetic market regimes
  • Social media trend analysis with sarcasm detection and language awareness

Notable Architectural Patterns

  • Service-oriented architecture: Separate services for sentiment, MARL, social media, exchanges, and evolutionary systems.
  • Repository pattern: SQLite adapter implementations for agent, tournament, sentiment, social, and backtest data access.

Features & Capabilities

Core Sentiment Analysis (Phase 1)

Sentiment Scoring Engine

  • 4-mode local analyzer: BASIC, ADVANCED, TRADING_SIGNALS, SMART modes with lexicon-based scoring.
  • Claude API fallback: Graceful degradation to local scoring when API is unavailable or rate-limited.
  • Cached sentiment results: 24-hour TTL per symbol; bulk refresh via authenticated API endpoint.
  • Bull/Neutral/Bear classification: Composite sentiment with confidence scoring and reasoning summaries.

Market Data & Content Aggregation

  • Live market data via CoinGecko: OHLCV history, price, market cap, 24h volume for 50+ coins.
  • News aggregation: NewsAPI integration with source ranking and relevance filtering.
  • Content signal pipeline: Reddit, Twitter/X-ready scoring with sarcasm detection and context extraction.
  • Optional FinBERT scoring: Remote Hugging Face inference endpoint for financial sentiment classification.
  • Multi-language support: Language detection and potential translation routing (infrastructure ready).

Dashboard & User Interface

  • React-based dashboard: 7-tab interface with real-time polling (10-min default) and SSE-based live updates.
  • Sentiment Lab: Analyze individual symbols, view cached sentiment, load rankings, inspect mode metadata.
  • Global health pill: 30-second polling of system health with expandable service status view.
  • Responsive design: Mobile-friendly with Chart.js visualizations for price and performance curves.
  • Dark mode ready: Theme infrastructure in place for light/dark toggle persistence.

Multi-Agent Reinforcement Learning (Phase 2)

MARL Tournament Engine

  • Shared order book matching: Price-time FIFO with realistic slippage modeling and commission deduction.
  • 3-agent risk profiles: AGGRESSIVE (5% risk/trade), CONSERVATIVE (1%), SCALPING (3%).
  • Tournament modes: SINGLE (one-shot), EVOLUTIONARY (mutation + replacement), CONTINUOUS (live learning).
  • Q-learning agents: 50-feature state space, 5-action policy (no action, buy small/large, sell small/large), epsilon-greedy exploration.
  • Experience replay: Prioritized buffer with configurable batch size and learning rates.
  • Persistent learning state: Agent Q-tables stored in SQLite; additive pre-training and live learning paths.

Agent Identity & Cosmetics

  • Agent registry: Unique identity, name, emoji, color, biography, and nickname.
  • Genealogy tracking: Parent/child relationships persisted; navigable lineage explorer in UI.
  • Win-rate leaderboard: Persistent ranking by cumulative profit, win %, and fitness score.
  • Competition history: Per-agent tournament participation with results, KPIs, and equity curves.
  • Agent retirement: Mark underperforming agents as inactive; exclude from breeding pools.

Evolutionary System

  • Multi-generation orchestration: Fitness evaluation → survival selection → crossover → mutation → repeat.
  • Fitness scoring: 0–100 composite metric balancing profit, Sharpe ratio, max drawdown, and win %.
  • Selection algorithms: Configurable survival percentage for tournament-style or elitism strategies.
  • Genetic crossover: UNIFORM (random gene swapping) or BLENDED (weighted averages).
  • Mutation engines: LIGHT, MEDIUM, HEAVY stochastic perturbation of genome parameters.
  • Genome persistence: SQLite CRUD for agent parameters; tournament results linked to generation.

Exchange Integration & Real Trading

  • Multi-provider support: Crypto.com (default), Binance.US, Coinbase Advanced Trade, Alpaca.
  • Trading modes: PAPER (in-memory simulation), SANDBOX (live quotes, no real orders), LIVE (real trading with safety guards).
  • Safety guardrails: Kill switch (max daily loss %), max open positions, position size cap, $1 minimum notional.
  • Credential storage: AES-256-GCM encryption for API keys; requires BROKER_MASTER_KEY env var.
  • Order management: Buy/sell limit orders with detailed fills, commission tracking, and slippage reporting.
  • Paper exchange: In-memory order book with synthetic pricing fallback for realistic simulation.

Backtesting & Pre-Training

  • Backtesting engine: Historical OHLCV simulation with KPI reporting (profit, Sharpe, max drawdown, win %).
  • Synthetic market generator: 5-regime price series (Bull trend, Bear trend, Sideways, Volatile crash, Volatile pump) for offline training.
  • Pre-trainer service: Run agents through configurable episodes/steps on synthetic markets before live competition.
  • Convergence tracking: Learning curves for policy value and loss metrics.
  • UI integration: Backtesting tab with agent config, run history, and reload of persisted results.

Architecture

System Architecture Overview

The project follows a service-oriented backend architecture with a React-based frontend communicating via REST APIs and Server-Sent Events.

Backend Architecture Layers

1. Express Application Layer (index.ts, routes/)

The main server bootstrap lives in backend/src/index.ts. It:

  • Initializes the Express app with CORS, Helmet, and body parsing middleware.
  • Connects to SQLite storage and waits for health confirmation before starting.
  • Registers routers for sentiment, MARL, social media, agents, evolutionary, and trading endpoints.
  • Sets up cron jobs for hourly social scraping, midnight metrics reset, and optional BullMQ worker initialization.
  • Implements graceful shutdown lifecycle (SIGTERM, SIGINT) to close database and queue connections.

Route routers live in backend/src/routes/:

  • marl-competition.ts → /api/marl/* (tournament CRUD, start, results, equity curves)
  • marl-real-trading.ts → /api/marl/broker/* (credential storage, balance, live orders)
  • social-media.ts → /api/social-media/* (refresh, trending, items, health)
  • agent-stats.ts → /api/agents/* (registry, leaderboard, customization)
  • evolutionary.ts → /api/evolutionary/* (breed, summary, genome, genealogy)
  • trading.ts → /api/trading/* (backtesting, pre-training)

2. Service Layer (services/)

Business logic is split into modular services:

  • SentimentService: Claude API wrapper with local fallback.
  • SentimentAnalyzerEngine: Local 4-mode NLP scoring (BASIC, ADVANCED, TRADING_SIGNALS, SMART).
  • ContentSignalService: NewsAPI + Reddit + X scoring pipeline.
  • MarlCompetitionEngine: 90KB tournament orchestration, agent state management, order book matching.
  • EvolutionaryOrchestrator: Multi-generation loop linking fitness, selection, crossover, mutation.
  • ExchangeFactory & TradingService: Provider routing (PAPER/SANDBOX/LIVE) with safety guards.
  • SocialMediaScraperManager: Orchestrates 7-platform scraper suite with fault tolerance.
  • BacktestingEngine: Historical simulation with KPI reporting.

3. Data Access Layer (repositories/)

Repository pattern abstracts database interactions:

  • SQLite adapter implementations: agent, backtest, broker, sentiment, social, tournament repositories.
  • Interface contracts: Defined in repositories/interfaces/; enables future swapping (PostgreSQL, Azure Tables).
  • Bulk operations: Transaction-based upserts for social items and trending snapshots.
  • Query patterns: Cursor-based pagination, TTL-based pruning, indexed lookups by symbol/date.

4. Storage & Persistence (storage.ts, database/)

SQLite singleton: WAL mode, foreign keys enabled, synchronous writes.

  • StorageService manages DB lifecycle and connection pooling.
  • SocialStore opens independent connection for social_media_items, trending_topics, trending_topic_history, source_metadata.
  • Schema migrations: SQL migrations for agent identity, evolutionary tables, broker credentials.
  • Encryption: AES-256-GCM for stored broker API keys (requires BROKER_MASTER_KEY).

5. Queue & Worker Infrastructure (queues/, workers/)

Optional distributed job processing via BullMQ + Redis:

  • TournamentQueue: Async SIMULATED tournament jobs; processed by tournament-worker-process.
  • ScraperQueue: Async social scraping jobs; processed by scraper-worker-process.
  • Fallback behavior: When REDIS_URL absent, tournaments use Worker Threads and scraper uses setImmediate (in-process).
  • Worker concurrency: Configurable via TOURNAMENT_WORKER_CONCURRENCY, SCRAPER_WORKER_CONCURRENCY.

6. Dependency Injection (container.ts, tsyringe)

Services registered in a tsyringe container for lifecycle management and testability.

Data Flow & Request Lifecycle

Sentiment Analysis Request

Frontend (Sentiment Lab) → POST /api/refresh-sentiment → SentimentService.analyzeSentiments() → Claude API (with local fallback) → SQLite cache (24h TTL) → Frontend update via polling or manual refresh

MARL Tournament Request

Frontend (MARL tab) → POST /api/marl/tournaments/start → MarlCompetitionEngine (backend thread or BullMQ worker) → Tournament execution: agents compete, Q-tables update, results persisted → GET /api/marl/tournaments/:id/equity-curve (polling) → SSE /api/marl/tournaments/:id/events (live updates) → Frontend renders live equity curves and trade feed

Social Scraping Request

Hourly cron or manual trigger → SocialMediaScraperManager orchestrates 7 scrapers → Items scored via 4-signal pipeline → Bulk upsert to SQLite social_media_items → TrendingDiscoveryEngine aggregates and calculates velocity → Frontend polls /api/social-media/trending and items endpoint → UI renders trending topics and feed with filters

Technology Decisions & Rationale

SQLite over PostgreSQL/MySQL: Simplifies deployment (single file), sufficient for current scale, WAL mode provides concurrency. Future migration to Postgres via repository abstraction is low-effort.

BullMQ + Redis (optional): Enables horizontal worker scaling for compute-heavy tournaments and scraping without blocking main API thread. Graceful degradation when absent.

Express over Next.js API routes: Explicit server control, easier middleware composition, clearer separation from frontend build tooling.

React + Vite over Next.js client: Lighter frontend, simpler API proxying, faster dev rebuild. No SSR complexity needed.

Chart.js over D3/Recharts: Lightweight, minimal dependencies, fast real-time equity curve rendering. D3 overkill for current use case.

Tsyringe over manual DI: Standardized container pattern, testable with mock injection, reduces boilerplate.

Strengths

✓ Architectural Maturity & Clean Layering

The service-oriented backend architecture demonstrates strong separation of concerns. Services (SentimentService, MarlCompetitionEngine, SocialMediaScraperManager, ExchangeFactory) are well-isolated, testable, and pluggable. The repository pattern decouples persistence from business logic, enabling future database swaps. The DI container (tsyringe) manages lifecycle cleanly. This is not a monolithic ball of mud—it's a thoughtful, layered design.

✓ Feature Completeness & Ambition

Three complete phases covering sentiment analysis, multi-agent RL, and social intelligence represent a significant amount of work. The combination of real-time market data, Claude-powered NLP, evolutionary trading agents, and cross-source social monitoring is non-trivial. Most features work end-to-end; live tournament monitoring via SSE, agent breeding UI, and trending discovery are polished and integrated. This is not a toy project.

✓ Zero-Trust & Security Posture

Broker credentials are AES-256-GCM encrypted with a master key (BROKER_MASTER_KEY). No plaintext secrets in code. The trading service wraps all exchanges with safety guards (kill switch, position limits, minimum notional). Risk management is thoughtful. This is a serious production mindset, not a rough prototype.

✓ Documentation & Onboarding

CLAUDE.md is exceptional. It provides clear command references, architecture diagrams (text-based), database schema notes, test patterns, and environment variable catalogs. README.md is comprehensive with quick-start, phase summaries, and deployment guidance. A new developer can spin up the app and understand data flow within hours. Few projects invest this level of care in developer documentation.

✓ Testing Culture & CI/CD

56–70% test coverage across backend and frontend is solid. GitHub Actions enforces docs validation, linting, type-checking, and testing before merging. Jest and Vitest are configured with sensible defaults. Postman collections provide API testing. This shows a team that values quality gates.

✓ Multi-Exchange Integration

Support for Crypto.com, Binance.US, Coinbase, and Alpaca with unified ExchangeInterface abstraction is mature. The ability to toggle trading providers and modes (PAPER/SANDBOX/LIVE) via config is excellent. Credential encryption and per-provider fee handling are professional touches.

Concerns & Areas for Improvement

⚠ Branch Coverage Gaps (40% backend, 51% frontend)

While line coverage is respectable (56–70%), branch coverage reveals significant untested conditional paths. Error handling, edge cases, and fallback logic are sparse. This increases the risk of subtle bugs under stress or unusual conditions (API timeouts, rate limits, malformed data). Recommendation: Prioritize error path testing (scraper fallbacks, exchange API degradation, agent Q-learning edge cases) before scaling to live trading or high-frequency tournaments.

⚠ MARL Competition Engine Complexity (840 LOC, 63% coverage)

The competition engine is the most critical and complex module. At 840 lines with 63% coverage, some order-matching logic, agent elimination rules, and tie-breaking scenarios may not be fully exercised. This is where financial behavior emerges—subtle bugs could lead to agent collusion, order manipulation, or unfair fitness calculations. Recommendation: Add property-based tests for order-matching invariants (e.g., total liquidity is never negative, price moves monotonically for the same pair) and tournament fairness (agents with identical genomes have same fitness distribution).

⚠ Database Schema Versioning & Migrations

Migrations exist for agent identity and evolutionary tables, but the migration system is not versioned or atomic. If a user runs on an old schema, silent data loss or constraint violations could occur. No explicit rollback strategy. Recommendation: Adopt numbered migration files (e.g., 001-initial.sql, 002-add-agent-identity.sql) with checksums and rollback tests. Consider using a lightweight migration runner (e.g., sql.js or a simple Node script) that tracks applied migrations.

⚠ Rate Limiting & API Quota Management

Social scrapers (Twitter, Reddit, YouTube, TikTok) have quota limits. Source metadata tracks error counts, but there's no proactive rate-limit detection or backoff strategy. If a quota is exceeded, the scraper may silently fail or degrade. The hourly cron could hit burst limits. Recommendation: Implement exponential backoff, track rate-limit headers (Retry-After, X-RateLimit-Remaining), and queue jobs for retry during low-traffic windows. Add alerts when quota is approaching.

⚠ Sentiment Analysis Caching & Cache Invalidation

Sentiment results are cached with a 24-hour TTL. But if sentiment for a coin changes dramatically (e.g., breaking news), users will see stale scores for up to 24 hours. The refresh endpoint requires API authentication, so dashboard users can't manually invalidate. Recommendation: Add a "clear cache for symbol" endpoint with lightweight auth (session-based, not API key). Consider shorter TTL (6 hours) during high-volatility windows, or trigger cache invalidation on significant price swings (>5% hourly change).

⚠ Social Scraper Coverage & Maintenance Burden

7 social scrapers are a lot to maintain. Twitter, Discord, Telegram, YouTube, TikTok all have evolving APIs and rate-limit policies. The code for each scraper is tightly coupled to platform specifics. If a platform changes its API (e.g., X restricts v2 access), the scraper breaks silently or returns empty results. Recommendation: Add integration tests with fixture data (saved HTML/JSON from each platform). Create a scraper health dashboard showing last-successful-run timestamps. Mark scrapers as "experimental" or "stable" based on test pass rates. Consider deprecating low-value scrapers (e.g., YouTube may have limited crypto discussion).

⚠ Live Trading Safety & Audit Trail

The trading service has safety guards (kill switch, position limits), but there's no audit trail of why orders were placed/rejected. If an agent makes a bad decision that costs money, there's no way to trace the decision path, state values, or Q-learning parameters at execution time. Recommendation: Log decision context before every order: state vector, Q-values, chosen action, market depth, current equity. Archive logs for post-mortems. Implement a "dry-run" mode that logs decisions without executing.

⚠ Agent Genealogy & Breeding Pool Explosion

The agent lineage tracking is clever, but there's no mechanism to prevent genealogy explosion. After 20 generations with 10 agents per generation, you could have 200+ agents in the registry. Breeding from this pool could lead to inbreeding (genetic bottleneck) if not careful. The UI shows genealogy trees, but the selection algorithm doesn't enforce diversity. Recommendation: Add diversity metrics to the fitness calculation (e.g., Hamming distance from population mean). Implement a "culling" strategy that removes low-fitness agents or those beyond a max age. Add telemetry for genealogy tree depth and branch factor.

⚠ Frontend SSE & Polling Fallback Robustness

Live tournament monitoring uses SSE with polling fallback. But the fallback logic could be more robust—if the SSE stream hangs (partial data), the poll might also hang. Network-level debugging is hard. No heartbeat/ping mechanism to detect stale connections. Recommendation: Add server-side heartbeat (comment line every 10s if no events). Implement client-side connection timeout (30s idle). Add telemetry tracking which path (SSE vs polling) is being used. Display connection status in UI (green for live, yellow for polling, red for offline).

⚠ Realistic Paper Trading Not Implemented

PAPER mode uses in-memory order matching. There's no realistic paper mode that applies real fees/slippage. This makes backtesting results optimistic—agents trained on PAPER might perform worse on SANDBOX/LIVE. (Issue #4–6 are marked "not started".) Recommendation: Implement REALISTIC_PAPER mode quickly if live trading is planned. Even if tournament scheduling (Issue #2) is deferred, fee/slippage modeling is important for realistic agent training.

Conclusion

It demonstrates strong software engineering practices (layered architecture, dependency injection, testing, documentation) and non-trivial domain expertise (MARL, evolutionary algorithms, sentiment analysis, exchange APIs).

The main gap is not architectural—it's edge-case resilience. Before production trading or high-frequency agent tournaments, the project would benefit from:

  • Increased branch coverage, especially for the MARL competition engine.
  • Audit trails and observability around financial decisions.
  • Robust handling of API failures and quota limits (especially for social scrapers).
  • Migration versioning and database recovery strategies.
